# os作品对比报告

## 一、对比对象说明

本次对比的新提交作品为基于对仓库 `tianjingongye`（根目录名称）的分析所得的推测作品，项目核心基于 **ArceOS** 开源项目进行深度定制和开发，是一个支持多硬件架构的类 Unix 操作系统内核。

本次对比使用的历史作品描述报告包括：
1. **历史作品报告1（nankai）**：基于 `nonix` 项目，一个使用 Rust 语言开发、支持多架构（RISC-V 64、x86_64、AArch64、LoongArch64）的操作系统内核，具备完整的进程、内存、文件和系统调用管理功能。
2. **历史作品报告2（qinghua）**：基于 **ArceOS** 框架开发的 unikernel（单内核）操作系统，同样支持多架构（x86_64、RISC-V 64、AArch64、LoongArch64），但采用 unikernel 架构，与传统多进程操作系统有本质区别。
3. **历史作品报告3（tianjindaxue）**：一个使用 Rust 语言开发的 **宏内核操作系统**，名为 **MonkeyOS**，支持多架构和多种运行平台，具备完善的进程、内存、文件系统和系统调用实现。

**说明**：本次对比基于三份历史作品报告。如果历史报告数量不足，会影响对比的全面性，但目前三份报告已覆盖不同的技术路线和实现风格，具有一定的代表性。

## 二、新提交作品概览

新提交作品（推测名为 `tianjingongye`）是一个基于 **ArceOS** 框架进行深度定制和开发的操作系统内核。

**仓库结构**：
- **`arceos/`**：核心操作系统框架，包含硬件抽象层（`axhal`）、内存管理（`axmm`）、文件系统（`axfs`）、网络协议栈（`axnet`）、任务管理（`axtask`）、系统调用接口（`arceos_posix_api`）以及用户库（`axlibc`、`axstd`）。
- **`api/`、`core/`**：比赛特定的核心逻辑层，提供任务管理、内存管理、资源管理和 futex 同步等核心抽象，以及更符合比赛要求的高级 API 实现。
- **`apps/`**：测试应用和评测脚本的集合。
- **`configs/`**：多架构配置文件。
- **`crates/`、`vendor/`**：自研驱动和第三方依赖缓存。
- **`src/`、`bin/`、`scripts/`、`docs/`**：顶层入口代码、构建脚本和文档。

**主要模块**：
- 启动模块、内核初始化模块、内存管理模块、进程/任务管理模块、中断/异常处理模块、系统调用模块、文件系统/驱动模块、构建与运行模块均**存在相关证据**，但部分模块（如进程模型）的具体实现方式尚不确定。

**当前完成度**：项目具备较高的完成度，拥有完整的启动、中断、系统调用、内存管理、文件系统、网络栈等核心模块，并实现了 `/proc/interrupts` 等高级特性。同时，项目正在从 Unikernel 向多进程演进，但进程模型的支持情况（是否真正实现用户态进程隔离）**无法从现有材料确定**。

## 三、历史作品概览

历史作品在结构、技术路线和核心模块方面具有以下共性：

**结构特点**：
- 多数作品采用 **分层架构**：内核源码（`os/src/` 或 `kernel/src/`）、用户程序（`user/` 或 `apps/`）、引导程序（`bootloader/` 或 `boot/`）、第三方依赖（`vendor/`）、文档（`doc/`）清晰分离。
- 普遍包含 **核心子系统** 目录：系统调用（`syscall/`）、任务管理（`task/` 或 `tasks/`）、内存管理（`mm/`）、文件系统（`fs/`）、中断/异常处理（`trap/`）、驱动（`drivers/` 或 `driver/`）。

**技术路线**：
- **主语言**：Rust 是主流选择，部分作品在底层启动代码中使用汇编，在文件系统绑定中使用 C 语言。
- **架构支持**：多架构（RISC-V 64、x86_64、AArch64、LoongArch64）是参赛作品的常见特征，部分作品还支持真实硬件（如树莓派、K210、CV1811H）。
- **架构模式**：历史作品涵盖了 **宏内核**（如 `MonkeyOS`）和 **unikernel**（如基于 ArceOS 的作品），以及具备完整进程模型的作品（如 `nonix`）。

**核心模块**：
- 所有历史作品都实现了启动、中断、系统调用、内存管理和进程/任务管理等操作系统核心功能。
- 文件系统方面，普遍支持 ext4（通过 lwext4_rust 绑定）和 FAT32。
- 部分作品实现了 `/proc` 文件系统、共享内存、信号处理等高级特性。

## 四、核心模块对比

### 1. 启动模块

- **新作品**：**证据充分**。在 `arceos/modules/axhal/src/platform/` 目录下，包含 AArch64、LoongArch64 等架构的 `boot.rs` 文件，以及为树莓派 4 编写的链式加载器。启动代码负责从 CPU 上电到跳转到 Rust 内核入口的完整流程（配置页表、启用 MMU、切换异常级别等）。
- **历史作品1（nankai）**：**证据充分**。包含 `bootloader/rustsbi-qemu.bin`（RISC-V SBI 引导）和 `vendor/polyhal-boot/src/arch/x86_64/multiboot.S`（x86 Multiboot 启动），支持多架构和多核 CPU 启动。
- **历史作品2（qinghua）**：**证据充分**。同样基于 ArceOS，架构与当前新作品高度相似，包含多架构的启动代码。
- **历史作品3（tianjindaxue/MonkeyOS）**：**证据充分**。通过 `vendor/polyhal-boot` 和多个链接脚本实现多架构启动，支持 Multiboot 和 AP 处理器激活。

**对比总结**：四个作品的启动模块均较为完整。新作品与历史作品2（qinghua）在启动代码上高度相似（都基于 ArceOS 的 `axhal` 模块），但与历史作品1和3的实现框架不同。

### 2. 内核初始化模块

- **新作品**：**证据存在但不够充分**。推测的核心初始化路径位于 `arceos/modules/axruntime/` 以及顶层 `src/main.rs`、`src/entry.rs`，负责堆初始化、中断控制器初始化、调度器初始化等。由于未详细阅读这些文件的源码，具体的初始化流程只能基于目录结构推测。
- **历史作品1（nankai）**：**证据充分**。`os/src/main.rs` 包含明确的内核入口和初始化流程，如控制台、日志系统、内存分配器、中断控制器、设备驱动、初始进程的创建。
- **历史作品2（qinghua）**：**证据存在**。同样基于 ArceOS，初始化逻辑在 `arceos/modules/axruntime/` 中，与当前新作品类似。
- **历史作品3（tianjindaxue/MonkeyOS）**：**证据充分**。`kernel/src/main.rs` 是内核主入口，包含硬件初始化、物理页分配器实现等。

**对比总结**：历史作品1和3的初始化逻辑更清晰、文档更充足。新作品和历史作品2的初始化模块分布较分散，可能更加模块化，但具体细节需要进一步分析源码。

### 3. 内存管理模块

- **新作品**：**证据存在**。体现在 `src/mm.rs`、`core/src/mm.rs`、`arceos/modules/axmm/` 以及 `vendor` 中的 `memory_set`、`memory_addr` 等 crate。支持页表管理、内存映射等功能，但具体实现细节（如是否支持 COW、懒加载、共享内存）**无法判断**。
- **历史作品1（nankai）**：**证据充分**。支持物理页帧分配、页表管理、懒加载、COW、共享内存、`mmap`/`munmap`，具备完整的用户堆和栈管理。
- **历史作品2（qinghua）**：**证据存在**。包含 `arceos/modules/page_table_multiarch/` 多架构页表实现和多种内存分配器，但基于 unikernel 架构，内存管理模式与传统操作系统不同。
- **历史作品3（tianjindaxue/MonkeyOS）**：**证据充分**。实现了完整的 `sys_brk`、`sys_mmap`（支持匿名、文件、共享映射，以及 MAP_FIXED、MAP_SHARED、MAP_ANONYMOUS 等标志）。

**对比总结**：历史作品1和3的内存管理实现更为完整和明确，特别是对 COW、共享内存、mmap 标志的支持。新作品虽然存在内存管理模块，但其功能范围和完成度**需要进一步确认**。

### 4. 进程或任务管理模块

- **新作品**：**证据存在，但实现模式不确定**。`core/src/task.rs` 和 `vendor/axprocess` 表明项目正在探索或实现了进程抽象。然而，`arceos/ulib/axstd/src/process.rs` 明确指出 ArceOS 是 unikernel，“没有进程概念”，`exit()` 直接关闭系统。这表明项目可能处于从 **unikernel 向多进程演进的过渡阶段**，真正的用户态进程隔离（`fork`/`exec`）**是否已稳定实现，无法确定**。
- **历史作品1（nankai）**：**证据充分**。实现了完整的进程生命周期管理，包括 `sys_fork`、`sys_execve`、`sys_exit`、`sys_wait4`，具备任务调度器、任务状态管理和进程组管理。
- **历史作品2（qinghua）**：**部分存在**。同样基于 ArceOS，其 `process.rs` 说明没有传统进程概念，但 `api/` 层提供了某种进程抽象供测试使用。**严格意义上，这是一个 unikernel，不是真正的多进程操作系统**。
- **历史作品3（tianjindaxue/MonkeyOS）**：**证据充分**。定义了完整的进程控制块（`ProcessControlBlock`）、线程控制块（`ThreadControlBlock`）和用户任务（`UserTask`），实现了 `sys_chdir`、`sys_getcwd`、`sys_exit`、`sys_execve` 等系统调用。

**对比总结**：这是新作品与历史作品差异最大的模块。历史作品1和3是传统的多进程操作系统；历史作品2与新作品同属于 ArceOS 家族，但新作品通过 `core/task.rs` 和 `axprocess` 表现出更强的向多进程演进意图。然而，其进程模型的成熟度和稳定性**无法从现有报告判断**。

### 5. 中断或异常处理模块

- **新作品**：**证据存在且完善**。体现在 `arceos/modules/axhal/src/arch/{aarch64, loongarch64, x86_64}/trap.rs` 和对应的汇编文件 `trap.S`。这是每个架构都会实现的关键模块，负责系统调用、页错误、外部中断等事件的处理。
- **历史作品1（nankai）**：**证据存在**。`os/src/trap/` 目录和 `polyhal-trap` 库提供了中断处理框架，支持用户态/内核态切换。
- **历史作品2（qinghua）**：**证据存在且完善**。同样基于 ArceOS，拥有与当前新作品相同的中断处理架构。
- **历史作品3（tianjindaxue/MonkeyOS）**：**证据存在**。`kernel/src/main.rs` 中提到的 `kernel_interrupt` 函数表明系统具备中断处理能力，但详细的实现分布在 `crates/` 子模块中。

**对比总结**：所有作品的中断/异常处理模块均比较成熟。新作品和历史作品2在这一模块上实现相似。

### 6. 系统调用模块

- **新作品**：**证据存在**。`src/syscall.rs`、`api/src/lib.rs` 和 `arceos/modules/arceos_posix_api/` 共同构成系统调用处理层，提供了 POSIX 接口实现（如 `open`、`read`、`write` 等）。
- **历史作品1（nankai）**：**证据充分**。`os/src/syscall/` 目录包含 `process.rs`、`mm.rs`、`fs.rs` 等文件，实现了进程控制、内存操作、文件操作等标准系统调用。
- **历史作品2（qinghua）**：**证据存在**。同样基于 ArceOS，拥有 `src/syscall.rs` 主入口和 `arceos/modules/arceos_posix_api/` 模块。
- **历史作品3（tianjindaxue/MonkeyOS）**：**证据充分**。`kernel/src/syscall/` 目录包含 `mm.rs`（内存相关）、`task.rs`（进程/任务相关）等文件，实现了与 Linux 兼容的系统调用接口。

**对比总结**：所有作品都实现了系统调用模块。新作品和历史作品2基于相同的 ArceOS POSIX API 框架，而以 `nonix` 和 `MonkeyOS` 为代表的历史作品则倾向于独立实现系统调用层。

### 7. 文件系统或驱动模块

- **新作品**：**证据存在**。`arceos/modules/axfs/` 提供核心文件系统框架，`vendor/lwext4_rust` 表明支持 ext4 文件系统，`arceos/modules/axfs_vfs/` 是 VFS 抽象层。驱动方面，`crates/` 下包含特定硬件的驱动（如龙芯 2K1000 网卡驱动），`arceos/modules/axdriver/` 是驱动框架。
- **历史作品1（nankai）**：**证据充分**。通过 `lwext4_rust/` 支持 ext4，`os/src/fs/` 提供文件系统接口封装，并支持 FAT32 和 ext4 两种文件系统镜像。
- **历史作品2（qinghua）**：**证据存在**。同样基于 ArceOS，拥有 `arceos/modules/lwext4_rust/` 和 VFS 抽象。
- **历史作品3（tianjindaxue/MonkeyOS）**：**证据充分**。拥有完整的文件系统栈：`filesystem/vfscore/`（VFS 抽象层）、`filesystem/devfs/`、`filesystem/procfs/`、`filesystem/ramfs/`，以及独立的驱动模块（`driver/` 目录）。

**对比总结**：新作品和历史作品2的文件系统实现基于 ArceOS 的 `axfs` 和 `lwext4_rust`，而历史作品1和3则提供了更独立、结构更清晰的文件系统和驱动分层。历史作品3的文件系统栈最为丰富。

### 8. 构建与运行模块

- **新作品**：**证据存在且强大**。提供 `Makefile`、`build.rs`、`build_img.sh`、`run_in_docker.sh` 等脚本，`configs/` 下有多架构配置文件，`.github/workflows/` 表明配置了 CI/CD 流程。
- **历史作品1（nankai）**：**证据存在**。包含 `Makefile` 和 `runall.sh`，指定 `rust-toolchain.toml`。
- **历史作品2（qinghua）**：**证据存在**。拥有 `arceos/scripts/`、`scripts/` 等构建脚本和 `configs/` 配置文件。
- **历史作品3（tianjindaxue/MonkeyOS）**：**证据存在**。使用 TypeScript（`scripts/.ts`）编写构建脚本，同时包含 Dockerfile。

**对比总结**：所有作品都具备较完善的构建与运行模块。新作品在 CI/CD、多架构配置和 Docker 支持方面表现得较为全面。

## 五、技术路线差异分析

| 维度 | 新作品（tianjingongye） | 历史作品1（nankai/nonix） | 历史作品2（qinghua） | 历史作品3（tianjindaxue/MonkeyOS） |
|------|------------------------|---------------------------|----------------------|--------------------------------------|
| **编程语言** | Rust（主）、C、汇编 | Rust（主）、汇编、C（FFI） | Rust（主）、汇编 | Rust（主）、汇编、TypeScript（构建） |
| **目录组织** | 双层结构：核心框架（arceos/）+ 比赛定制（api/、core/、apps/） | 单层结构：os/、user/、bootloader/、lwext4_rust/ | 双层结构：arceos/ + 比赛定制（api/、apps/、core/） | 单层结构：kernel/、crates/、driver/、filesystem/ |
| **模块划分** | 高度模块化，以 crates 粒度组织（如 axhal、axmm、axfs） | 以目录划分模块（syscall/、task/、mm/） | 高度模块化，与 ArceOS 一致 | 以目录划分模块，crates/ 目录子模块化 |
| **构建方式** | Makefile + 多架构配置（.toml）+ Docker + CI/CD | Makefile + rust-toolchain.toml | Makefile + 多架构配置 + Docker | TypeScript 脚本 + Dockerfile |

**核心差异**：
- **架构基础**：新作品和历史作品2均深度基于 **ArceOS 框架**，具备一致的模块划分和硬件抽象层设计；历史作品1和3则倾向于 **独立设计**，从底层自行实现核心功能。
- **模块化程度**：基于 ArceOS 的作品（新作品和历史作品2）模块化粒度更细，以独立的 Rust crate 组织（如 `axhal`、`axmm`），便于复用和替换；独立设计的作品则以功能目录划分，内聚性更强。
- **构建工具**：历史作品3使用了 TypeScript 编写构建脚本，这在操作系统开发中较为少见；基于 ArceOS 的作品则使用标准的 Makefile 和配置文件。
- **进程模型**：这是最关键的差异点。历史作品1和3是传统的多进程操作系统；历史作品2是典型的 unikernel；新作品则表现出从 unikernel 向多进程演进的趋势，但目前处于过渡状态。

## 六、完成度与复杂度对比

**代码结构完整性**：
- 新作品和历史作品2（ArceOS 系）结构完整，模块划分清晰，且拥有多架构配置和真实硬件支持（树莓派、龙芯 2K1000）。
- 历史作品1和3同样结构完整，但模块的组织方式不同（非 ArceOS 分层方式）。

**模块覆盖范围**：
- 四个作品均覆盖了启动、中断、系统调用、内存管理、进程/任务管理、文件系统/驱动等核心模块。
- 新作品和历史作品2额外支持网络协议栈（`axnet`、`smoltcp`）。
- 历史作品1和3在进程管理、内存管理（COW、共享内存）方面的实现更为明确和完整。

**工程组织复杂度**：
- 新作品和历史作品2（ArceOS 系）的工程复杂度较高，需要协调多个层级的 crate（`arceos/` 框架 + `api/`、`core/`、`apps/` 等定制层），以及多架构配置和真实硬件支持。
- 历史作品1和3的工程组织相对简单，依赖关系更清晰，但同样支持多架构和真实硬件。

**对比总结**：
- 新作品在 **多架构支持、真实硬件适配、网络协议栈** 方面表现突出。
- 历史作品1和3在 **传统操作系统功能的完成度**（如进程模型、COW 内存管理）上更具优势。
- 新作品如果成功实现从 unikernel 到多进程的演进，其完成度和复杂度将显著提升。

## 七、可能创新点分析

基于现有材料，新作品相比历史作品可能具有以下创新点：

1. **从 Unikernel 向多进程的演进路径**：与历史作品2（纯 ArceOS unikernel）不同，新作品通过 `core/task.rs`、`axprocess` 等组件，表现出主动突破 unikernel 局限、引入进程抽象的努力。这种 **框架内的架构转型** 是一个值得关注的创新举措，但该工作是否已经成功完成 **无法从已有材料确认**。

2. **多架构 + 真实硬件的深度适配**：新作品在支持四种主流架构（x86_64、AArch64、RISC-V、LoongArch64）的同时，还专门为树莓派 4 和龙芯 2K1000 编写了驱动和启动加载器。这种对真实物理机的深度支持，在历史作品中并不普遍（历史作品3同样支持 K210 等开发板，但程度未知）。

3. **模块化的 fs/proc 文件系统实现**：新作品实现了 `/proc/interrupts` 虚拟文件系统，通过 `proc_interrupts.rs` 暴露中断统计信息。这种模块化的 `/proc` 实现方式，相比直接在系统调用中硬编码，更具可维护性和可扩展性。

## 九、总结评价

### 新作品水平
新提交作品（`tianjingongye`）目前处于一个 **较高但仍在演进中的水平**。它基于成熟的 ArceOS 框架，具备完整的启动、中断、系统调用、内存管理、文件系统、网络栈等核心模块，多架构和真实硬件支持出色，工程组织规范、文档齐全。然而，其最关键的差异——**从 unikernel 向多进程的演进**，目前仅停留在存在相关组件的阶段，具体完成度未知。

### 与历史作品的主要差异
- **技术路线**：与历史作品1（`nonix`）和3（`MonkeyOS`）等独立设计的宏内核相比，新作品基于 ArceOS 框架，模块化程度更高，支持更丰富的特性和硬件。
- **进程模型**：与历史作品2（纯 unikernel）相比，新作品表现出明显的向多进程演进趋势，这是最大的差异化特点。
- **功能覆盖**：在网络协议栈、真实硬件驱动方面，新作品比历史作品1和3更全面；但在传统操作系统功能的明确性和完成度上（如 `fork`/`exec`、COW、共享内存），历史作品1和3目前更具优势。

### 后续补强建议
1. **确认并完善进程模型**：这是最核心的补强方向。需要明确当前 `core/task.rs`、`axprocess` 等组件是否真正实现了用户态进程隔离和上下文切换，并完善 `fork`、`execve`、`wait4` 等关键系统调用。
2. **增强内存管理功能**：参考历史作品1和3，补充 COW（写时复制）、懒加载、共享内存等高级内存管理特性，提升对主流 Linux 测试套件的兼容性。
3. **扩展系统调用覆盖**：当前 POSIX 调用覆盖可能不够全面（如信号处理、进程组管理、管道等），需要对照 Linux 测试标准（如 LTP）查漏补缺。
4. **加强测试与文档**：完善评测脚本和调试工具，确保演进后的功能稳定可靠；补充设计文档，特别是 unikernel 向多进程演进的设计决策和实现细节。
5. **性能优化**：如果进入更高级别的比赛，需要关注系统调用延迟、内存分配效率、中断处理性能等关键指标。

# 源码智能分析系统 \- 源码分析文档

项目版本：v6.3
AI 后端：DeepSeek API（deepseek\-chat）

## 1\. 项目源码结构

源码智能分析系统是一个面向操作系统竞赛项目的自动分析与评价平台，采用分层模块化设计，共 28 个核心模块，实现从源码读取到最终评估报告的全流程自动化分析。

### 1\.1 设计目标与核心价值

**设计目标**：为操作系统竞赛项目提供客观、全面、智能的评估能力，解决传统人工评审效率低、主观性强、标准不统一的问题。

**核心价值**：

|价值|说明|
|---|---|
|提高评审效率|自动化分析几分钟完成人工几小时的工作|
|保证评审客观性|统一标准和算法，减少主观因素影响|
|提供多维度洞察|从代码结构、模块依赖、技术难度等多维度分析|
|支持横向对比|与历史项目对比，给出相对位置和改进建议|

**设计原则**：

- **分层解耦**：每个层次职责明确，通过标准化接口交互

- **双轨策略**：核心环节采用主方案\+兜底方案，保证鲁棒性

- **缓存优先**：支持断点续跑和智能跳过，提高重复运行效率

- **容错降级**：异常情况自动降级，不影响整体流程

### 1\.2 整体架构

系统分为八个层次，从下到上依次为：


**数据流向**：源码输入 → 代码解析 → AI 函数理解 → 调用图构建 → 模块画像 → 仓库画像 → 历史检索对比 → 五维评分 → 最终报告

## 2. 核心模块说明

### 2.1 代码解析模块（ast_parser + code_splitter）

**核心功能**：将原始源代码转换为结构化的代码块，是后续所有分析的基础。

**技术特点**：
- **双引擎解析**：优先使用 Tree-sitter 进行精确的 AST 解析，解析失败时自动回退到正则表达式切分
- **多语言支持**：支持 Rust、C、C++、Python、Bash、汇编等多种编程语言
- **智能过滤**：自动过滤重复代码块、过短的类型引用、函数内部的嵌套定义
- **丰富的代码块类型**：支持函数、结构体、枚举、Trait、Impl 块、常量、静态变量、宏等多种代码单元

**输出数据结构**：
```json
{
  "block_id": "block_1",
  "file_path": "kernel/src/syscall/fs.rs",
  "file_score": 0.85,
  "language": "rust",
  "parser": "tree_sitter",
  "start_line": 41,
  "end_line": 102,
  "type": "function",
  "name": "do_openat",
  "content": "fn do_openat(...) { ... }"
}
```

### 2.2 AI 函数理解模块（code_understander）

**核心功能**：利用大语言模型对每个代码块进行深度语义分析，理解函数的功能、逻辑、调用关系和所属模块。

**技术特点**：
- **六维分析**：功能摘要、执行逻辑拆解、调用关系提取、模块分类标注、分析依据说明、不确定性标注
- **并发处理**：使用 ThreadPoolExecutor 支持多线程并发分析，提高处理效率
- **断点续跑**：支持进度保存和恢复，中断后可从上次位置继续分析
- **异常降级**：AI 返回非标准 JSON 时，自动将原始内容保存到 summary 字段并标记异常
- **Prompt 工程**：精心设计的 Prompt 引导 AI 输出结构化 JSON，包含明确的字段要求和约束

**输出数据结构**：
```json
{
  "block_id": "block_1",
  "name": "do_openat",
  "summary": "实现 openat 系统调用，根据路径、目录文件描述符和标志打开或创建文件/目录",
  "logic_steps": ["步骤1...", "步骤2..."],
  "called_functions": ["path_lookup", "file_alloc", "fd_install"],
  "related_os_module": "filesystem",
  "evidence": "函数中包含文件路径解析、文件描述符分配等典型文件系统操作",
  "uncertainty": "部分内部函数实现细节需结合上下文进一步确认"
}
```

### 2.3 调用图构建模块（call_graph_builder）

**核心功能**：构建函数级别的调用关系图谱，支持内部调用与外部调用的区分。

**技术特点**：
- **双轨提取**：AI 语义提取 + 正则规则提取，两种方式互补
- **置信度机制**：单源提取置信度 0.6-0.7，双源验证置信度提升至 0.9
- **调用分类**：自动区分内部调用（项目内函数）和外部调用（库函数/系统调用）
- **函数名标准化**：处理 `module::func`、`object.method` 等多种调用形式，统一为函数名
- **模块级统计**：按模块统计函数数量、调用边数、内/外部调用比例

**输出数据结构**：
```json
{
  "repo_name": "zhengzhoudaxue111",
  "node_count": 20,
  "edge_count": 415,
  "internal_edge_count": 7,
  "external_edge_count": 408,
  "nodes": [...],
  "edges": [
    {
      "edge_id": "edge_1",
      "caller": "do_openat",
      "callee": "path_lookup",
      "call_type": "internal",
      "source": ["ai", "regex"],
      "confidence": 0.9
    }
  ],
  "module_stats": {...}
}
```

### 2.4 模块画像生成模块（module_summarizer）

**核心功能**：将函数级分析结果聚合为模块级画像，实现从微观到宏观的抽象。

**技术特点**：
- **AI 摘要生成**：调用大模型对模块内所有函数进行总结，生成模块级描述
- **核心函数筛选**：按输出调用数量排序，自动识别模块中的核心函数
- **权重计算**：综合函数数量（60%）和调用边数量（40%）计算模块权重
- **内聚耦合分析**：统计内部调用比例（内聚性）和外部调用比例（耦合性）
- **双版本输出**：精简版（AI 摘要）+ 完整版（统计指标 + 核心函数详情）

**输出数据结构**：
```json
{
  "module_name": "filesystem",
  "function_count": 6,
  "outgoing_edge_count": 177,
  "internal_call_ratio": 0.0169,
  "external_call_ratio": 0.9831,
  "module_weight": 0.3506,
  "core_functions": [
    {
      "name": "do_openat",
      "outgoing_call_count": 77,
      "summary": "实现 openat 系统调用..."
    }
  ],
  "module_summary": "该模块主要负责文件系统相关的系统调用实现...",
  "key_functions": [...],
  "call_flow_summary": "..."
}
```

### 2.5 混合检索模块（hybrid_retriever）

**核心功能**：结合结构化检索和语义检索，从历史项目库中找出与目标项目最相似的项目。

**技术特点**：
- **双路检索**：结构化检索（关注指标匹配）+ RAG 语义检索（关注描述相似）
- **权重融合**：结构化权重 0.65，语义权重 0.35，综合计算混合相似度
- **排名基语义分**：使用 rank-based score 替代原始距离分数，更稳定可靠
- **自动 Query 生成**：根据目标项目画像自动生成 RAG 检索查询语句
- **证据留存**：保存语义检索的文档证据，支持结果溯源

**检索权重配置**：
- 结构化检索权重：0.65
- 语义检索权重：0.35
- RAG 返回数量：10
- 最终返回数量：5

### 2.6 五维评分模块（score_evaluator）

**核心功能**：对目标 OS 项目进行五个维度的结构化评分，提供全面的质量评估。

**技术特点**：
- **五维评分模型**：原创性、新颖性、可实践性、技术难度、完成度，每维满分 20 分，总分 100 分
- **双轨评分机制**：规则参考分（基于结构化指标）+ AI 评分（基于语义理解）
- **偏差约束**：AI 评分与规则参考分的偏差原则上不超过 ±3 分，保证评分稳定性
- **兜底机制**：AI 评分失败时，自动使用规则参考分作为兜底
- **置信度标注**：每个评分结果附带置信度等级（high/medium/low）
- **改进建议**：自动生成项目的优势、不足和改进建议

**评分等级划分**：
- excellent：≥ 85 分
- good：70 - 84 分
- medium：55 - 69 分
- weak：< 55 分

## 3\. 代码解析方法

### 3\.1 解析流程

```
原始源码文件
    ↓
文件类型检测 → 判断编程语言
    ↓
Tree-sitter AST 解析 → 生成语法树
    ↓
代码块提取 → 遍历 AST 提取函数/类/结构体等
    ↓
后处理 → 去重、过滤、名称提取
    ↓
[成功?] ──是──→ 输出 Tree-sitter 解析结果
    │
    否
    ↓
正则表达式回退 → 使用正则模式匹配
    ↓
输出正则切分结果
```

**设计原则**：优先高质量解析，保证可用性；核心路径用最好方案，边缘情况用兜底方案。

### 3\.2 语言检测与处理

|扩展名|识别语言|Tree\-sitter 支持|处理方式|
|---|---|---|---|
|`.rs`|rust|是|Tree\-sitter 解析|
|`.c`, `.h`|c|是|Tree\-sitter 解析|
|`.cpp`, `.hpp`|cpp|是|Tree\-sitter 解析|
|`.py`|python|是|Tree\-sitter 解析|
|`.sh`|bash|是|Tree\-sitter 解析|
|`.S`, `.s`, `.asm`|assembly|否|正则切分|
|`Makefile`, `Cargo.toml`|config|否|整体配置块|

### 3\.3 Tree\-sitter AST 解析

**解析方式**：

- 使用本地预编译 grammar 包，离线可用，版本稳定

- 每种语言创建独立解析器实例，可复用

- 深度优先遍历（DFS）整个语法树

- 按节点类型筛选目标代码块

**目标节点类型**：

- Rust：function\_item、struct\_item、enum\_item、trait\_item、impl\_item、const\_item、static\_item、macro\_definition

- C/C\+\+：function\_definition、struct\_specifier、enum\_specifier、type\_definition

**名称提取策略**：

- 优先使用 `child_by_field_name("name")` 获取命名字段（更稳定）

- 备选方案：查找子节点中的 identifier / type\_identifier

- 不同语言采用不同策略，适配各自 AST 结构

### 3\.4 后处理与过滤

**通用后处理**：

- 按起止行排序

- 按 `(type, name, start_line, end_line)` 去重

**C/C\+\+ 特殊过滤**：

- 过滤函数内部的嵌套定义（内部 struct/enum/typedef 是实现细节）

- 过滤无意义的类型引用（前向声明、只有一行的结构体声明等）

- 使用 `is_inside_function()` 判断节点是否位于函数内部

### 3\.5 正则回退机制

**触发条件**：Tree\-sitter 库未安装、语言不支持、解析异常、解析结果为空

**正则切分流程**：

1. 正则匹配代码块起始位置（覆盖多种语法形式）

2. 括号匹配器确定结束位置

3. 处理字符串、注释中的括号（避免误匹配）

4. 合并排序、去重

**括号匹配器**：支持单行注释 `//`、块注释 `/* */`、转义字符，通过深度计数找到匹配的右大括号。

### 3\.6 配置文件处理

对于 Makefile、Cargo\.toml、链接脚本等配置文件，整体作为一个配置代码块：

- 类型标记为 config

- 名称为文件名

- 内容为完整文件内容

**统一输出格式**：无论使用哪种解析方式，最终输出格式统一，后续模块无需关心具体解析方式。

### 3\.7 设计思路与技术亮点

**设计思路**：

- 优先高质量解析，保证核心路径的解析质量

- 兜底方案保证系统鲁棒性，边缘情况也能正常工作

- 统一输出格式，实现解析层与上层模块的完全解耦

- 针对不同语言特点做特殊优化，不是一刀切的通用方案

**技术亮点**：

- 本地预编译 grammar 包，离线可用，版本稳定

- 深度优先遍历 AST，按节点类型精准筛选

- C/C\+\+ 特殊处理，过滤函数内部嵌套定义

- 括号匹配器支持注释、字符串、转义字符，准确率高

- 多种语言统一接口，上层模块无需关心语言差异

## 4\. Tree\-sitter 的作用

### 4\.1 Tree\-sitter 简介

Tree\-sitter 是一个开源的增量解析器生成器和解析库，能够为多种编程语言快速生成准确的抽象语法树（AST）。

**核心优势**：

- **高性能**：C 语言编写，解析速度极快，支持增量解析

- **多语言支持**：官方和社区提供丰富的语言 grammar 包

- **容错性强**：即使代码有语法错误，也能生成部分有效的语法树

- **离线可用**：grammar 预编译为本地库，无需网络连接

### 4\.2 核心作用

**精准的语法解析**：

- 基于语法规则精确匹配，天然支持任意深度嵌套

- 正确区分代码、注释和字符串

- 新增语言只需添加 grammar 包

- 相比正则表达式：不会误匹配、能处理嵌套结构、不受注释/字符串干扰

**多语言统一解析框架**：

- 支持 Rust、C、C\+\+、Python、Bash 五种语言

- 每种语言独立 grammar 包，统一解析接口

- 上层模块无需关心具体语言的语法差异

- 不同语言项目可用同一套分析流程，保证结果一致性和可比性

**丰富的代码单元提取**：

- 不仅能提取函数，还能提取结构体、枚举、Trait、Impl 块、常量、静态变量、宏等

- 为后续 AI 分析提供更丰富的素材

- 能全面理解代码结构，而不仅限于函数层面

### 4\.3 集成方式

**本地 grammar 包**：使用预编译的 tree\_sitter\_rust、tree\_sitter\_c、tree\_sitter\_cpp、tree\_sitter\_python、tree\_sitter\_bash，保证离线可用、版本稳定、启动快速。

**解析器管理**：
每种语言创建独立解析器实例，设置好 language 后可重复使用。
```python
def get_parser_for_language(language):
    parser = Parser()
    if language == "rust":
        parser.language = Language(tree_sitter_rust.language())
        return parser
    # ... 其他语言
```

**AST 遍历与提取**：

- 深度优先遍历访问每个节点

- 通过字节偏移量从源码提取对应文本（确保多字节字符准确）

- 按字段名查找子节点（`child_by_field_name`），比按索引查找更可靠

### 4\.4 双引擎策略

**策略**：优先 Tree\-sitter，回退正则

**触发回退的情况**：

- Tree\-sitter 库未安装

- 文件语言不支持（如汇编语言）

- 解析过程中发生异常

- 解析结果为空（无有效代码块）

**优势**：

- 保证主流语言的解析质量

- 保证系统的兼容性和鲁棒性

- 可交叉验证，两种方式都提取到的代码块置信度更高

### 4\.5 技术价值与意义

**对系统的价值**：

- Tree\-sitter 是整个系统的技术基石，没有它就没有高质量的代码解析

- 多语言支持让系统能分析不同语言的 OS 项目，适用范围更广

- 精准的语法解析为后续 AI 分析提供了高质量的输入

- 丰富的代码单元提取让系统不仅能分析函数，还能理解数据结构等

**相比传统方案的优势**：

- 比正则表达式更准确、更可靠

- 比自研解析器成本更低、支持语言更多

- 比 LSP 方案更轻量、无需启动语言服务器

- 离线可用，不依赖网络连接

## 5\. 函数切块方法

### 5\.1 切块概述

函数切块（Code Splitting）是将完整的源代码文件按语法结构切分为独立代码块的过程。每个代码块对应一个独立的语法单元，如函数、结构体、枚举等。

**为什么需要切块**：

- 大模型上下文窗口有限，无法处理整个项目

- 函数级粒度大小适中，语义完整，适合 AI 单次分析

- 函数之间有调用关系，便于构建调用图

**切块粒度**：函数级为主，其他定义（结构体、枚举、常量等）为辅。

### 5\.2 Tree\-sitter 切块流程

1. **解析源码生成 AST**：Tree\-sitter 将源码解析为抽象语法树

2. **遍历 AST 筛选目标节点**：深度优先遍历，筛选需要保存的代码块节点

3. **提取节点信息**：代码内容、起止行号、节点类型、名称等

4. **节点类型映射**：将 Tree\-sitter 节点类型映射为系统内部统一类型

**质量优势**：基于语法规则，不会像正则那样容易出错，边界准确，类型丰富。

### 5\.3 正则切块流程（回退方案）

1. **正则匹配起始位置**：匹配多种语法形式的函数定义（普通函数、pub 函数、async 函数、unsafe 函数等）

2. **括号匹配确定结束位置**：

    - 从左大括号开始逐字符扫描

    - 遇到 `{` 深度\+1，遇到 `}` 深度\-1

    - 深度回到 0 时找到匹配的右大括号

    - 处理字符串、注释、转义字符中的括号

3. **合并排序去重**

**定位**：作为回退方案，保证 Tree\-sitter 不可用时系统仍能工作。

### 5\.4 切块后处理

- **去重**：按 `(类型, 名称, 起始行, 结束行)` 去重

- **C/C\+\+ 特殊过滤**：过滤函数内部嵌套定义、过滤无意义的类型引用

- **完整性检查**：确保每个代码块是完整的语法单元

- **大小合理性检查**：过滤异常大小的代码块

### 5\.5 输出格式

```json
{
  "repo_name": "项目名称",
  "repo_path": "仓库本地路径",
  "created_at": "创建时间",
  "block_count": 代码块总数,
  "blocks": [
    {
      "block_id": "block_1",
      "file_path": "源文件路径",
      "file_score": 文件重要性评分,
      "language": "编程语言",
      "parser": "解析器（tree_sitter/regex）",
      "start_line": 起始行号,
      "end_line": 结束行号,
      "type": "代码块类型",
      "name": "代码块名称",
      "content": "代码内容"
    }
  ]
}
```

### 5\.6 设计思路与技术亮点

**设计思路**：

- 函数级粒度是 AI 分析的黄金分割点，语义完整且大小适中

- Tree\-sitter 主方案保证质量，正则回退保证鲁棒性

- 统一输出格式，上层模块无需关心具体切块方式

- 后处理过滤保证输出质量，去掉冗余和噪声

**技术亮点**：

- 基于语法规则的精准切块，边界准确无误

- 支持多种代码单元类型，不仅限于函数

- 括号匹配器处理注释、字符串、转义字符

- C/C\+\+ 特殊过滤，去掉函数内部嵌套定义

## 6\. AI 函数理解方法

### 6\.1 概述

AI 函数理解是系统的智能核心层，利用大语言模型对每个代码块进行深度语义分析，将代码从"看不懂"变成"读得懂"。

**输入**：结构化的代码块
**输出**：带有语义标注的函数分析结果（功能摘要、执行逻辑、调用关系、所属模块等）

### 6\.2 分析维度

|维度|字段名|说明|
|---|---|---|
|功能摘要|`summary`|一句话概括函数主要作用|
|执行逻辑|`logic_steps`|拆解函数主要执行步骤|
|调用关系|`called_functions`|函数内部调用的其他函数|
|模块分类|`related_os_module`|所属操作系统模块|
|分析依据|`evidence`|判断的代码依据|
|不确定性|`uncertainty`|信息不足时的说明|

**支持的 OS 模块**：boot、memory、process、scheduler、interrupt、syscall、filesystem、driver、build、unknown

### 6\.3 Prompt 设计

**结构化 Prompt 包含**：

- 角色设定

- 上下文信息（仓库名称、代码块信息）

- 代码内容

- 分析要求（六个维度）

- 输出格式要求（合法 JSON）

**质量约束**：

- 必须基于代码内容分析，不要只根据函数名猜测

- 不要编造代码里没有体现的功能

- 无法判断模块就写 unknown

- 输出必须是合法 JSON，不要输出 Markdown

**预设模块列表的作用**：保证分类一致性，便于后续按模块聚合，减少 AI 自由发挥空间。

### 6\.4 并发与断点续跑

**并发分析**：

- 使用 ThreadPoolExecutor 实现多线程并发

- 可配置 max\_workers（默认 3）、save\_every（默认 5）、max\_blocks

- 并发数根据 API 速率限制调整，避免触发限流

- IO 密集型任务，多线程效率提升明显

**断点续跑**：

- 独立进度文件保存分析进度，每分析完一批写入一次

- 续跑时加载已分析的 block\_id 集合，跳过已分析的

- 使用 block\_id 作为唯一标识，确保不重复

- 支持网络断开、API 限流、程序崩溃、手动停止等各种中断场景

### 6\.5 异常处理与降级

**JSON 解析失败时的降级策略**：

- 原始 AI 回复保存到 summary 字段

- 其他数组字段设为空列表

- 模块分类设为 unknown

- evidence 和 uncertainty 中说明情况

**并发场景容错**：

- 每个块的分析用 try\-except 包裹

- 单个代码块失败不影响整体流程

- 失败的块会被标记，但不会导致整个流程崩溃

### 6\.6 设计思路与技术亮点

**设计思路**：

- 结构化输出设计，便于后续程序处理和分析

- 预设模块分类，保证分类一致性和可比性

- 质量约束机制，防止 AI 编造和过度发挥

- 进度文件独立保存，支持断点续跑

**技术亮点**：

- 六大分析维度全面覆盖函数理解的各个方面

- ThreadPoolExecutor 并发分析，充分利用 API 能力

- 异常降级机制，保证系统稳定性

- 断点续跑支持，不浪费已完成的工作

## 7\. 调用图构建方法

### 7\.1 概述

调用图（Call Graph）是表示函数之间调用关系的有向图，节点代表函数，边代表调用关系。调用图是理解代码结构、分析模块依赖、评估项目复杂度的重要基础。

**本系统特点**：AI 语义提取 \+ 正则规则提取的双轨策略，结合两者优势生成高质量调用图。

### 7\.2 双轨提取策略

|提取方式|优点|缺点|置信度|
|---|---|---|---|
|AI 语义提取|能理解语义，识别隐式调用（回调、动态分发）|可能漏提取，速度慢，成本高|0\.6\-0\.7|
|正则规则提取|速度快，成本低，确定性强|只能匹配显式调用，可能误匹配|0\.6\-0\.7|
|**双源验证**|两种方式都提取到，更可靠|\-|**0\.9**|

**AI 提取**：从 function\_analysis 的 called\_functions 字段获取，来源标记为 "ai"

**正则提取**：直接扫描函数源码，用正则匹配函数调用模式，过滤控制语句（if、for、while、switch 等），来源标记为 "regex"

### 7\.3 合并与增强

**合并去重**：按 \(caller, callee\) 对分组，同一对有多个来源则合并 source 列表

**置信度规则**：

- 单源（只有 ai 或只有 regex）：置信度 0\.6\-0\.7

- 双源（ai \+ regex 都提取到）：置信度提升至 0\.9

**函数名标准化**：

- 直接调用 `func()` → `func`

- 模块路径 `module::func()` → `func`

- 方法调用 `obj.method()` → `method`

- 链式调用 `a.b().c()` → `b`, `c`

**目的**：统一函数名表示，便于与节点名称匹配，减少重复边。

### 7\.4 调用分类

|调用类型|说明|判断依据|
|---|---|---|
|内部调用（internal）|调用项目内定义的函数|callee 在调用图节点集合中|
|外部调用（external）|调用库函数、系统调用或未定义的函数|callee 不在节点集合中|

**意义**：

- 内部调用反映项目内部的模块依赖关系

- 外部调用反映项目对外部库和系统的依赖

- 内部调用比例反映项目内聚性

- 外部调用比例反映项目对外部的依赖程度

### 7\.5 节点信息增强与模块统计

**节点元数据**：name、normalized\_name、file\_path、language、start\_line、end\_line、module、summary

| 字段 | 说明 | 来源 |
|------|------|------|
| `name` | 函数名 | 代码切块 |
| `normalized_name` | 标准化函数名 | 名称标准化 |
| `file_path` | 源文件路径 | 代码切块 |
| `language` | 编程语言 | 代码切块 |
| `start_line` | 起始行号 | 代码切块 |
| `end_line` | 结束行号 | 代码切块 |
| `module` | 所属模块 | AI 函数分析 |
| `summary` | 功能摘要 | AI 函数分析 |

**价值**：调用图不仅是拓扑结构，还包含丰富的语义信息，每个节点都带有功能说明和所属模块。

**模块统计**：按模块统计函数数量、输出调用边数、内部调用边数、外部调用边数，为模块画像提供数据基础。

**输出版本**：

- 基础版（call\_graph）：基于正则提取，生成快

- 增强版（call\_graph\_enhanced）：AI \+ 正则双轨合并，更完整，置信度更高

### 7\.6 设计思路与技术亮点

**设计思路**：

- 双轨提取取长补短，结合 AI 语义理解和正则确定性匹配

- 置信度机制让下游模块可以根据可靠性加权使用

- 函数名标准化保证不同调用形式能正确匹配

- 节点信息增强让调用图不仅是拓扑结构，还包含语义信息

**技术亮点**：

- AI \+ 正则双轨提取，交叉验证提高置信度

- 自动区分内部调用与外部调用，便于分析模块依赖

- 函数名标准化处理，统一不同调用形式

- 节点元数据丰富，调用图自带语义信息

## 8\. 模块画像生成方法

### 8\.1 概述

模块画像是对操作系统项目中各个功能模块的结构化描述，包括模块的功能、核心函数、调用关系、权重、完成度等信息。

**定位**：连接微观函数分析和宏观项目评估的关键中间层

**为什么需要模块画像**：

- 函数级分析太细，几百个函数很难直接理解整体结构

- 项目级分析太粗，不够具体

- 模块级是中间的黄金分割点

### 8\.2 模块分类

**分类来源**：AI 函数分析的 related\_os\_module 字段

**常见 OS 模块类型**：

|模块名|说明|典型函数|
|---|---|---|
|`boot`|启动引导|入口函数、初始化函数|
|`memory`|内存管理|内存分配、页表管理|
|`process`|进程管理|进程创建、上下文切换|
|`scheduler`|调度器|调度算法、时间片管理|
|`interrupt`|中断处理|中断处理函数、异常处理|
|`syscall`|系统调用|系统调用入口、分发函数|
|`filesystem`|文件系统|文件操作、目录操作|
|`driver`|设备驱动|驱动初始化、IO 操作|
|`network`|网络协议栈|网络协议、套接字|
|`build`|构建系统|构建脚本、配置|
|`unknown`|未分类|无法判断模块的函数|

### 8\.3 AI 模块摘要生成

**输入**：模块名称、模块内所有函数的分析结果、模块相关的调用关系

**输出**：

- `module_summary`：模块总体作用总结

- `key_functions`：关键函数及其角色

- `call_flow_summary`：模块内部调用关系或执行流程总结

- `evidence`：支持判断的证据

- `uncertainty`：不确定信息说明

**优先级**：优先分析非 unknown 模块，unknown 模块优先级最低

### 8\.4 模块统计指标

**规模指标**：

- `function_count`：模块内函数数量

- `file_count`：涉及的源文件数量

- `files`：涉及的源文件列表

**调用关系指标**：

- `outgoing_edge_count`：输出调用边总数

- `internal_edge_count`：内部调用边数

- `external_edge_count`：外部调用边数

- `internal_call_ratio`：内部调用比例（内聚性）

- `external_call_ratio`：外部调用比例（耦合性）

**权重指标**：

- `module_weight`：模块权重（0\-1），函数数量占比 × 0\.6 \+ 调用边占比 × 0\.4

### 8\.5 核心函数筛选与版本

**核心函数筛选策略**：

- 按输出调用数量排序（调用其他函数越多的通常越核心）

- 优先保留有 AI 摘要的函数

- 每个模块最多保留 10 个核心函数

**两个版本**：

- **精简版（module\_summary）**：AI 生成的语义摘要，适合快速浏览

- **完整版（module\_summary\_full）**：语义摘要 \+ 统计指标 \+ 核心函数详情，适合深度分析

### 8\.6 仓库画像聚合

**仓库画像（Repo Profile）**：项目级的更高层级抽象

**内容包括**：

- 项目基本信息

- 函数分析摘要

- 调用图摘要

- 模块摘要

- 项目类型推断

- 结构复杂度估算

- 核心模块选择

- 模块完成度估算

**项目类型推断**：
系统根据模块分布自动推断 OS 项目类型：

| 项目类型 | 判断条件 |
|---------|---------|
| `teaching_os_or_general_kernel` | 有 memory + process + syscall + filesystem + driver |
| `minimal_general_kernel` | 有 memory + process + syscall + scheduler/interrupt |
| `basic_kernel` | 有 memory + process + syscall |
| `rtos_like_kernel` | 有 scheduler + driver，没有 filesystem |
| `filesystem_focused_project` | 有 filesystem，没有 process |
| `network_or_driver_focused_kernel` | 有 network + driver |
| `memory_management_focused_project` | 有 memory，没有 process |
| `unknown_os_project` | 以上都不匹配 |

**模块完成度估算**：
根据函数数量、调用边数、内部调用数、文件数估算模块完成度：

| 指标 | 高分条件 | 权重 |
|------|---------|------|
| 函数数量 | ≥ 20 个 | 35% |
| 输出调用边数 | ≥ 30 条 | 25% |
| 内部调用边数 | ≥ 15 条 | 25% |
| 文件数量 | ≥ 5 个 | 15% |

### 8\.7 设计思路与技术亮点

**设计思路**：

- 从微观到宏观的多层抽象，既能深入细节又能把握整体

- 模块权重算法综合考虑函数数量和调用边数，更科学合理

- 核心函数筛选基于调用数，符合"调用越多越核心"的直觉

- 双版本设计满足不同场景需求，精简版快速浏览，完整版深度分析

**技术亮点**：

- 智能模块分类，基于 AI 语义理解自动归类

- 丰富的统计指标，量化评估模块规模和复杂度

- 核心函数自动筛选，快速定位关键代码

- 仓库画像聚合，形成项目级整体描述

## 10\. 系统使用方式

### 10\.1 概述

系统采用命令行菜单式交互设计，通过 `python main.py` 启动后进入交互界面，输入命令执行对应功能。

### 10\.2 命令体系

系统共 32 个命令，分为四大类：

|类别|数量|说明|
|---|---|---|
|主流程命令|15个|核心分析流程，从入库到评估的完整链路|
|调试命令|5个|单步调试，用于开发和问题排查|
|Legacy 旧版命令|9个|早期版本保留的兼容命令|

**常用命令**：

- `final_analyze_hybrid`：一键端到端分析（推荐，混合检索模式）

- `final_analyze`：一键端到端分析（标准模式）

- `batch_ingest_history`：批量入库历史项目

- `help`：查看所有可用命令

### 10\.3 两种演示模式

|模式|适用场景|特点|
|---|---|---|
|分步骤演示|调试、教学、问题排查|可观察每个步骤输出，便于理解工作原理|
|一键演示|快速评估、比赛演示、批量处理|一条命令完成所有分析，简单高效|

**一键命令**：

- `final_analyze`：标准模式（基于结构化检索）

- `final_analyze_hybrid`：混合检索模式（结构化 \+ RAG 语义，推荐）

- 提高重复运行效率，避免重复计算

- 基于文件存在性判断，简单可靠

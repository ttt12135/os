## 五、程序运行流程推测

基于 xv6 原始框架和当前代码，该项目的运行流程大致如下：

1.  **启动 (Bootloader/SBI)**: 硬件平台（K210）上电后，由硬件固化或片上 bootloader 跳转到内核的汇编入口点 (例如 `entry_k210.S`)。对于 QEMU 或 VisionFive2，OpenSBI 会将 CPU 引导至 S-mode，然后跳转到内核入口。
2.  **内核早期初始化 (entry.S)**: 进行启动CPU的准备工作：设置栈指针 (`sp`)，清除 BSS 段，为内核运行提供干净的 C 环境。
3.  **C 语言环境初始化 (start.c / main.c)**: 调用 `main()` 函数，该函数会：
    - 初始化控制台 (`consoleinit`)，打印启动信息。
    - 初始化物理内存分配器 (`kinit`)。
    - 初始化虚拟内存系统 (`kvminit`, `kvmminithart`)，建立内核页表。
    - 初始化进程表和相关锁 (`procinit`)。
    - 初始化陷阱处理（中断向量）(`trapinithart`)。
    - 初始化设备中断控制器（如 `plicinit`）和设备驱动。
    - 初始化文件系统（`fat32_init` 或 `ext4_init`），挂载 SD 卡。
4.  **创建第一个用户进程**: 调用 `userinit()` 或 `initcode` 机制，将根文件系统上的 `init` 程序（通常是 `sh` 或自定义的 `init`）加载到内存中。
5.  **启动调度器 (scheduler)**: `main()` 最后调用 `scheduler()` 函数，该函数进入无限循环，寻找 `RUNNABLE` 状态的进程并切换过去执行。
6.  **用户态运行与系统调用**: 调度器切换到 `init` 进程，进程在用户态执行。当进程发起 `ecall` 指令时，CPU 陷入 `kernelvec`，保存上下文到 `trapframe`，跳转到 `usertrap()` 处理。`usertrap` 识别出系统调用陷阱，调用 `syscall()` 分发到具体函数（如 `sys_open`， `sys_fork`），执行完毕后，通过 `usertrapret()` 恢复用户态上下文并 `sret` 回到用户程序继续执行。
7.  **时钟中断与调度**: 时钟中断发生时，CPU同样陷入 `kerneltrap` 或 `usertrap`。在时钟中断处理函数中，会调用 `proc_on_timer_tick()`，该函数会更新进程的时间片计数器，并可能触发 **MLFQ** 或 **优先级** 等调度算法的重新评估，最终决定是否调用 `yield()` 来切换进程。
